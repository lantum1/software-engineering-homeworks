# Проектирование документной модели MongoDB

## Какие коллекции были созданы

В рамках семантики сервиса File Management Service используется три основные коллекции:

1) folders -  хранит информацию о папках пользователей
2) files - хранит информацию о файлах пользователей и ссылки на принадлежащую папку и бинарные данные файлов
3) fs.files и fs.chunks - служебные коллекции GridFS для хранения бинарного содержимого файлов (создаются и управляются драйвером MongoDB автоматически)

## Структура каждой коллекции

### Коллекция folders
#### Описание коллекции
Содержит информацию о папках, созданных пользователями

#### Пример документа в коллекции
```
{
  _id: UUID('fd99d646-6ae3-42e3-85ca-5271700ea04c'),
  userId: UUID('8584e317-cd96-4504-b8ce-2bbd774cf3b2'),
  name: 'Рабочие документы',
  metadata: {
    created: ISODate('2026-03-13T12:45:02.946Z'),
    updated: ISODate('2026-04-07T12:45:02.946Z')
  }
}
```

#### Описание полей документа в коллекции
1) _id - UUID - ID папки
2) userId - UUID - ID пользователя-владельца папки
3) name - string - имя папки
4) metadata.created - ISODate - таймстемп создания папки
5) metadata.updated - ISODate - таймстемп обновления метаданных папки

#### Индексы в коллекции
Команды в mongosh для создания индексов:
```
db.folders.createIndex({ userId: 1, name: 1 }, { unique: true })
```

1. Индекс нужен для ускорения поиска папки по имени папки и по ID пользователя. Префиксная часть индекса используется для получения всех имен папок конкретного пользователя. Весь индекс используется для быстрой проверки наличия у данного пользователя папки с существующим именем. Кроме этого, благодаря unique служит защитой - запрещает создавать две папки с одинаковым именем у одного пользователя.

### Коллекция files
#### Описание
Содержит метаданные файлов, привязанных к конкретным папкам и пользователям

#### Пример документа в коллекции
```
{
  _id: UUID('dbf3f1c5-aa12-4ca1-af24-2be6bfba402a'),
  folderId: UUID('fd99d646-6ae3-42e3-85ca-5271700ea04c'),
  userId: UUID('8584e317-cd96-4504-b8ce-2bbd774cf3b2'),
  name: 'Техническое_задание.docx',
  metadata: {
    mimeType: 'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
    size: Long('1548672'),
    checksum: 'sha256:003633decd8882da56689ebb48e230bfafea9f48568375d34f2ad64ef5638874',
    created: ISODate('2026-04-05T12:45:02.948Z'),
    updated: ISODate('2026-04-08T12:45:02.948Z')
  },
  gridFsFileId: ObjectId('69db93ce656f58a9e33d88bf')
}
```

#### Описание полей документа в коллекции
1) _id - UUID - ID файла
2) folderId - UUID - ID папки, в которой лежит файл
3) userId - UUID - ID пользователя-владельца файла
4) name - string - имя файла с расширением
5) metadata.mimeType - string - MIME-тип контента. Используется для заголовка Content-Type при отдаче
6) metadata.size - int64 - размер файла в байтах
7) metadata.checksum - string - SHA-256 хэш содержимого
8) metadata.created - ISODate - таймстемп создания файла
9) metadata.updated - ISODate - таймстемп обновления метаданных файла
10) gridFsFileId - ObjectId - ссылка на документ в служебной коллекции fs.files. Через нее MongoDB агрегирует чанки из fs.chunks

#### Индексы в коллекции
Команды в mongosh для создания индексов:
```
db.files.createIndex({ folderId: 1, name: 1 }, { unique: true })
db.files.createIndex({ folderId: 1, userId: 1 });
```

1. Индекс нужен для быстрого поиска файла с определенным именем в определенной папке. Кроме этого, благодаря unique служит защитой - запрещает создавать два файла с одинаковым именем в одной папке.
2. Индекс нужен для быстрого поиска всех файлов в определенной папке у определенного пользователя. Также ускорит будущие запросы только по folderId, если они появятся. 

### Коллекции GridFS fs.files и fs.chunks:
#### Описание
Стандартные коллекции драйвера MongoDB. Не требуют ручной настройки индексов или структуры. Поле gridFsFileId из коллекции files ссылается на _id документа в fs.files, который агрегирует чанки из fs.chunks.

## Обоснование выбора стратегии Embedded / Reference

1. Связь файлов и папок друг с другом
Файлы и папки - отдельные коллекции files и folders соответственно. Связь между файлами и папками - **reference** через folderId поле. folderId ссылается на поле _id конкретного документа в коллекции folders. Встраивание файлов в документ коллекции folders со временем нарушит лимит MongoDB в 16 МБ на одно сообщение и поставит ограничение на количество файлов в папке.
2. Метаданные папки
На данный момент, к метаданным папки относятся два поля - таймстемпы created и updated. Связь между папками и их метаданными через **embedded** документ metadata. Даты создания и изменения папки всегда читаются и обновляются вместе с родительской сущностью. Вынос в поддокумент metadata улучшает логическую структуру схемы, позволяет атомарно обновлять поля metadata вместе с обновлением информации о самой папке (например, при изменении имени папки).
3. Метаданные файла
На данный момент, к метаданным относятся следующие поля - mimeType, size, checksum, created, updated. Связь между файлами и их метаданными через **embedded** документ metadata. Эти данные логически принадлежат файлу - в частности, mimeType требуется при каждом запросе на скачивание для формирования HTTP-заголовка. Также встраивание исключает лишние запросы к документам, позволяет атомарно обновлять поля created и updated поддокумента metadata вместе с обновлением информации о самом файле (например, при изменении имени файла).
4. Бинарное содержимое файла
MongoDB имеет ограничение в 16 МБ на размер документа. GridFS автоматически разбивает файлы на чанки, поддерживает потоковую передачу и не нагружает коллекцию метаданных. Поэтому, само бинарное содержимое файла и информация о файле связаны между собой через **reference** через gridFsFileId.
