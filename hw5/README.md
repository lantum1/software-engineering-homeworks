# Домашнее задание 05: Оптимизация производительности через кеширование и rate limiting

## Описание изменений

Сначала, были определны эндпоинты, которые подлежали кешированию и Rate Limiter'у. Описание проектирования кеширования и Rate Limiter'а представлено в [этом файле](perofmance_design.md).

### Кеширование
Было добавлено кеширование полностью согласно описанию в файле. В docker-compose был добавлен Redis контейнер с версией из примера в репозитории с домашками:
```
  redis:
    container_name: redis
    image: redis:6.2-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    networks:
      - maxdisk-network
    restart: unless-stopped
    command: redis-server --save 20 1 --loglevel warning
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
```
Впоследствии, Redis был добавлен в сервисы Identity Service (для эндпоинта GET /v1/users/search/login) и File Management Service (для эндпоинта GET /v1/folders).

С точки зрения реализации кеширования было выбрано кеширование на сервисном слое сервисов. Рассматривался выбор между кешированием на сервисном слое и кешированием на уровне отдельных запросов в БД, но у кеширования на уровне отдельных запросов в БД есть ряд неприятных недостатков:
1. Кеш не понимает семантику данных - отсутствие бизнес-контекста. Один и тот же запрос может выполняться из разных методов сервиса, следовательно сложно отследить зависимости
2. Сложность инвалидации - к примеру, на запрос "создание папки" приходилось бы инвалидировать все кеши условно SELECT запросов, то есть много каких ключей пришлось бы инвалидировать

Кроме этого, кеширование на уровне сервисного слоя еще и проще в реализации. Поэтому, было выбрано кеширование на сервисном слое - то есть результат выполнения бизнес-логики.


### Rate Limiter
Был добавлен Rate Limiter для двух эндпоинтов - для POST /v1/users/auth/login и POST /v1/users/auth/register. Сама стратегия Rate Limiter'а для данных эндпоинтов в реализации полностью соответствует описанию из файла. 

Из всего списка эндпоинтов, подлежащих Rate Limiter'у (а он был спроектирован для всех эндпоинтов), я выбрал именно эти два, потому что считаю их наиболее критическими для добавления Rate Limiter в первую очередь. 

Данные эндпоинты не имеют токена авторизационного, поэтому идентифицировать атакующего будет проблематично. Кроме этого, эти эндпоинты связаны с самой секьюрной частью системы - авторизацией/аутентификацией и регистрацией. Я посчитал, что в первую очередь необходимо ограничить brute-force для логина и ботовую регистрацию пользователей.

Сама реализация Rate Limiter'ов была выполнена на основе Redis кеша и запросов к нему - главная причина, почему не был выбран In Memory вариант, это потому, что система спроектирована как набор микросервисов. Каждый микросервис может масштабироваться горизонтально на несколько реплик. При In Memory варианте каждый инстанс считал бы запросы независимо от другого. С Redis все инстансы используют общее состояние.

Для синхронизации Rate Limiter'а не был использован PostgreSQL, потому что Redis значительно быстрее PostgreSQL - а Rate Limiter должен быть настолько быстрым, насколько возможно. Кроме этого, для информации по Rate Limiterингу нет требований по сохранности данных при сбоях - нет смысла использовать долговременное надежнок хранилище в виде PostgreSQL.

Для проверки работоспособности Rate Limiter был сделан [sh скрипт теста](identity-service/tests/rate_limiter_test.sh), проверяющий, что после N запросов к методам /v1/users/auth/login и /v1/users/auth/register начинает отдаваться HTTP код 429 с заголовкам X-RateLimit-Limit, X-RateLimit-
Remaining, X-RateLimit-Reset.

## Анализ производительности
### Как кеширование улучшает производительность
1. Снижение нагрузки на базу данных
Для примера с эндпоинтом просмотра списка папок пользователя GET /v1/folders

Пользователь открывает список папок 10 раз за минуту - 10 запросов к БД без кеширования. С кешированием 1 запрос к БД + 9 из кеша. Если пользователей 1000, то без кеширования это 10000 запросов к БД. С кешированием 1000 запросов к БД. На 90% снизили нагрузку на БД

2. Уменьшение RT кешируемых эндпоинтов
Для расчета RT брался **медианный** RT из 10 одинаковых запросов

2.1. GET /v1/folders
* Без кеширования 6 мс
* С кешированием 4 мс

2.2 GET /v1/users/search/login
* Без кеширования 25 мс
* С кешированием 5 мс

С учетом небольшого наполнения БД, выгода от добавления кеширования на эндпоинт GET /v1/folders на данный момент малозаметна - RT меняется в рамках погрешности. Кроме этого, на малое уменьшение RT влияет то, что эндпоинт GET /v1/folders обращается к MongoDB, которая тоже частично хранит в ОЗУ данные.

А что касается эндпоинта GET /v1/users/search/login, то там идет обращение к PostgreSQL - и несмотря на тоже малую заполненгость БД, выгода от добавления кеширования сильно более заметна (в 5 раз упал медианный RT)

### Как Rate Limiter улучшает производительность
1. Защита от DDoS. Без Rate Limiter, при относительно продолжительной большой нагрузки вероятность падения сервиса/сервисов растет - возможна каскадная деградация. После добавления Rate Limiter, особенно если на все эндпоинты, то после N запросов остальные запросы "в окне" станут отбрасываться - то есть Fail Fast, не дойдя до БД/сторонних интеграций и не заполняя очередь запросов сервиса
2. Предсказуемость ресурсов. С добавлением потолка RPS по эндпоинтам гораздо проще спрогнозировать максимально возможное потребление ресурсов сервисами

### Метрики для мониторинга производительности
#### Метрики кеширования
1. Hit Rate - доля запросов, которые были успешно обслужены из кеша, без обращения к базе данных. Чем выше, тем лучше. Собирать через кастомный каунтер в приложении и Prometheus
2. Miss Rate - доля запросов, которые не были найдены в кеше и потребовали обращения к базе данных. Метрика, обратная к Hit Rate
3. Cache Memory Usage - процент использованной памяти Redis относительно максимального лимита. Оптимальное значение: 30-70%. INFO memory в Redis
4. Eviction Rate - количество ключей, которые были принудительно удалены из кеша Redis из-за нехватки памяти. Чем ниже, тем лучше. INFO stats в Redis
5. Redis Latency - среднее время выполнения команд в Redis. Критично для кеширования и Rate Limiter'а: если Redis отвечает медленно, замедляется весь сервис. Чем ниже, тем лучше. Мониторить через SLOWLOG или Redis Exporter.

#### Метрики Rate Limiter
1. Rate Limited Requests - количество запросов, ограниченных Rate Limiter'ом за определенный период времени. Можно собирать через мониторинг HTTP кодов ответов 429. Если значение высокое, то либо слишком маленький лимит, либо атака. Если значение низкое, то либо слишком высокий лимит, либо все хорошо
2. Rejection Rate by Endpoint - доля запросов, отклоненных Rate Limiter'ом, в разрезе конкретного эндпоинта. Можно посчитать, зная метрику выше. Позволяет точечно понять, на какие эндпоинты была атака или где надо увеличивать/уменьшать лимит

#### Общие метрики производительности
1. RT - время ответа эндпоинтов сервиса по различным перцентилям. Чем ниже, тем лучше. Метрика стандартно экспортируется приложением в Prometheus
2. RPS - кол-во запросов на конкретный эндпоинт в секунду. Позволяет понять, когда сколько было запросов и на какие эндпоинты. Метрика стандартно экспортируется приложением в Prometheus
3. Потребление CPU контейнером - процент использования выделенных ЦПУ контейнером. При высоком проценте необходимо искать причину высокого потребления или увеличивать ресурсы. Метрика стандартно экспортируется приложением в Prometheus
4. Потребление ОЗУ контейнером - процент использования выделенной ОЗУ контейнером. При высоком проценте еобходимо искать причину высокого потребления или увеличивать ресурсы. Метрика стандартно экспортируется приложением в Prometheus
5. Database Connections - кол-во активных соединений к БД. Позволяет оценивать, достаточно ли коннектов к БД, если используется пул соединений на стороне приложения. Или позволяет оценивать, насколько много коннектов необходимо приложению в моменте. Позволяет выявить утечки соединений, если их не закрывать. Пул соединений, если используется, обычно сам экспортирует метрики в Prometheus
6. Network I/O - объем данных, передаваемых через сеть контейнера. Резкий рост может указывать на аномальную активность или неоптимальные ответы (например, отдача больших данных без пагинации). Экспортируется через cAdvisor/Prometheus.
7. Disk I/O - операции чтения/записи на диск. Критично для эндпоинтов загрузки/скачивания файлов. Высокая задержка диска может стать узким местом. Мониторить через cAdvisor или метрики хранилища (если используется внешнее, например, S3).

### Как измерить эффективность кеширования (hit rate)
#### Формула расчета
Hit Rate = Cache Hits / (Cache Hits + Cache Misses) * 100

Пример:
```
За период наблюдения:
- Запросов к кешу: 10000
- Попаданий (Cache Hits): 8500
- Промахов (Cache Misses): 1500

Hit Rate = 8500 / (8500 + 1500) * 100 = 85%
```

#### Как измерить
1. Добавить в приложение счетчики для подсчета хитов и миссов по каждому кешируемому эндпоинту, сколько раз значение было получено из кеша, а сколько раз был промах получения из кеша
2. Подключить Prometheus к приложению
3. Добавить экспортирование счетчиков в Prometheus
4. Подключить Grafana
5. Настроить дашборд по метрикам из Prometheus в Grafan'е
6. Наблюдать за дашбордом при типичном использовании системы (т.е. не пытаться самому воспроизводить Use Case'ы, которые в обынчой ситуации обычный пользователь не воспроизведет ненамеренно)
