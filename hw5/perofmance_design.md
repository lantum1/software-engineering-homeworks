# Описание стратегии кеширования и rate limiting
## Описание стратегии кеширования
### Выбор стратегии кеширования

Для данного проекта для всех кешируемых данных выбрана стратегия Cache Aside, так как его достаточно просто реализовать. Данный вариант отказоустойчив -  если кеш упал, приложение продолжает работать напрямую с БД. Кеш содержит только реально запрашиваемые данные - нет мусора. В этом варианте логика кеширования инкапсулирована в сервисном слое, не требует изменения в БД или создания дополнительнных слоев в сервисах (в сравнении с Read-Through).

Стратегия Write-Behind может привести к потере данных в БД, а в Write-Through, по моему мнению, мало смысла, потому что в выбранных эндпоинтах нет критически важных данных (например, финансовых транзакций).

### Выбор кешируемых эндпоинтов и TTL

1. GET /v1/folders
* TTL - 5 минут
* Список папок меняется редко (создание/удаление не каждую минуту). При этом пользователь может открывать список много раз за сессию. 5 минут - баланс между актуальностью и нагрузкой на БД. Основная актуальность обеспечивается инвалидацией при записи
* Стратегии инвалидации - при создании папки POST /v1/folders и при удалении папки DELETE /v1/folders/{id}
2. GET /v1/users/search/login
* TTL - 30 минут
* Профиль пользователя (имя, фамилия, email) меняется крайне редко. Поиск по логину используется для отображения информации о пользователе, где 30-минутная задержка некритична.
* Стратегии инвалидации - при обновлении профиля. Пока что такой функциональности нет, поэтому данный кеш инвалидируется только по TTL

### Почему остальные эндпоинты не кешируются

1. POST /v1/users/auth/login - логин пользователя. Проверка пароля должна быть всегда актуальной. Кеширование ответа может привести к аутентификации по устаревшим данным (например, после смены пароля)
2. POST /v1/users/auth/register - регистрация пользователя. Создает данные, а не читает. Кешировать нечего
3. POST /v1/folders - создаёт папку. После создания инвалидирует кеш списка папок, но сам ответ не кешируется
4. DELETE /v1/folders/{id} -  удаляет папку. Удаляет данные, после удаления инвалидирует кеш списка папок, но сам ответ не кешируется
5. POST /v1/folders/{id}/files - загрузка файлов. Тело запроса слишком велико для Redis, плюс однократная операция в рамках одного файла
6. GET /v1/folders/{id}/files?name= - скачивание файлов. Кеширование бинарных данных в Redis неэффективно. Для этого существует CDN или HTTP-кеширование
7. GET /v1/users/search - поиск пользователей по маске имени и фамилии. Не кешируем, потому что маски имени и фамилии создают огромное пространство ключей. HIT Rate будет низким, кеш быстро заполнится неперспективными записями

## Описание стратегии Rate Limiter'а
### Выбор эндпоинтов для Rate Limiting и алгоритмов

1. POST /v1/users/auth/login
* Алгоритм - Fixed Window Counter. Потому что данный алгоритм обеспечивает полную блокировку до окончания окна.
* Лимит - 10 запросов в минуту в рамках одного login из тела запроса

Полная блокировка после 10 попыток в минуту - для защиты от brute-force.\
Почему 10 запросов в минуту? Такого лимита достаточно для обеспечения защиты от brute-force.
```
Пароль из 8 символов (a-z, 0-9) = 36^8 = 2.8 триллиона комбинаций
При 10 попытках/минуту = 14400 попыток/день
Время подбора = 2.8×10^12/14400 = 194 миллиона дней
Вывод: 10/минуту делает brute-force практически невозможным
```

Данный эндпоинт не требует авторизационного JWT токена, поэтому это еще одна причина, почему он попадает под Rate Limiter

2. POST /v1/users/auth/register
* Алгоритм - Token Bucket. Сглаживание всплесков, плавное восстановление лимита
* Лимит - 1000 запросов в минуту глобально. Достаточно для легитимного трафика

Данный эндпоинт я предлагаю подвести под Rate Limiter, чтобы ограничить ботов, которые могут создавать кучу аккаунтов. Кроме этого, он не требует авторизационного JWT токена, поэтому это еще одна причина, почему он попадает под Rate Limiter

3. POST /v1/folders/{id}/files
* Алгоритм - Leaking Bucket. Сглаживание нагрузки на обработку файлов
* Лимит - 30 запросов в минуту для одного пользователя с userId из токена. Загрузка файлов не настолько частая редкая операция, чтобы лимита 30 запросов в минуту было недостаточно одному пользователю

Данный эндпоинт я предлагаю подвести под Rate Limiter, потому что загрузка файлов - тяжеловесная операция.

4. GET /v1/users/search
* Алгоритм - Fixed Window Counter
* Лимит - 60 запросов в минуту для одного пользователя с userId из токена - поиск не должен использоваться чаще, чем 1 раз в секунду

Данный эндпоинт я предлагаю подвести под Rate Limiter, потому что поиск по маске имени и фамилии дорогостоящая операция для БД (из-за операторов ILIKE). Fixed Window Counter алгоритм обеспечивает более жесткий лимит в минуту, чем Leaking Bucket или Token Bucket. Жесткий лимит защищает БД от злоупотреблений лучше

5. Остальные эндпоинты
* Алгоритм - Fixed Window Counter
* Лимит - 100 запросов в минуту для одного пользователя с userId из токена

Для всех остальных эндпоинтов я предлагаю ввести Rate Limiter с идентичными парамаметрами, потому что среди остальных эндпоинтов отсутствуют эндпоинты со сложными вычислениями или которые требовалось бы защитить с точки зрения ИБ (как для эндпоинтов register и login, которые без токена работают). Данные базовый лимит отсекает экстремальные сценарии: зацикленный скрипт, злонамеренная атака. 100 запросов в минуту достаточно высокий порог, чтобы не мешать легитимному использованию, но достаточный для ограничения ущерба
